### <a name="_3bfxc9a45514"></a>**Функциональные требования**
Решение развивает архитектуру депозитного MVP из `Task3` и добавляет канал публикации ставок для внутреннего кол-центра и внешнего партнёра. В фокусе только консультации по текущим депозитным ставкам; персонализированные и специальные условия остаются в контуре банка и не передаются партнёру.

|**№**|**Действующие лица или системы**|**Use Case**|**Описание**|
| :-: | :- | :- | :- |
| 1 | Сотрудник депозитного бэк-офиса, сотрудник кредитного бэк-офиса, АБС | Ведение ставок в мастер-системе | 1. Сотрудники депозитного и кредитного бэк-офиса ведут актуальные ставки и правила специальных условий в АБС.<br>2. АБС сохраняет промышленный реестр ставок вместо XLS.<br>3. `ABS Adapter` выгружает в онлайн-контур только актуальные публичные ставки, пригодные для консультаций. |
| 2 | `Rate Publication Service`, система кол-центра банка, оператор кол-центра | Просмотр актуальных ставок оператором кол-центра банка | 1. Система кол-центра запрашивает актуальные ставки через внутренний REST API `Rate Publication Service`.<br>2. | 3 | `Partner Export Job`, партнёрский кол-центр | Получение файла ставок партнёрским кол-центром | 1. `Partner Export Job` формирует файл с публичными депозитными ставками из `Deposit DB`.<br>2. Файл шифруется, версионируется и отправляется по согласованному почтовому каналу партнёру.<br>3. Партнёрский кол-центр загружает файл в свою систему и использует его для консультаций. |
| 4 | Оператор кол-центра банка, партнёрский оператор, клиент | Консультация клиента по ставкам | 1. Клиент звонит в кол-центр банка или партнёру.<br>2. Оператор открывает актуальный срез ставок в своей системе.<br>3. Оператор консультирует клиента по текущим условиям.<br>4. Если нужен продукт со специальными условиями, оператор переводит сценарий в стандартный процесс банка без самостоятельного расчёта ставки. |

### <a name="_u8xz25hbrgql"></a>**Нефункциональные требования**
В таблицу включены только требования, значимые для кейса передачи ставок в кол-центр.

|**№**|**Требование**|
| :-: | :- |
| 1 | АБС остаётся мастер-системой по депозитным ставкам; прямой онлайн-доступ системы кол-центра и внешнего партнёра к АБС запрещён. |
| 2 | Новая публикация ставок должна переиспользовать read model из `Task3` и не создавать дополнительную заметную нагрузку на Oracle БД АБС. |
| 3 | Внутренний кол-центр должен получать актуальные ставки автоматически, без ручной пересылки XLS, писем или звонков в бэк-офис. |
| 4 | Партнёрский кол-центр может работать только через файловый обмен; API-вызовы и SFTP недоступны. |
| 5 | Во внешний файл могут попадать только публичные ставки и служебные реквизиты версии выгрузки; персонализированные и специальные условия передавать нельзя. |
| 6 | Передача ставок партнёру должна быть защищённой и трассируемой: нужен журнал выгрузок, версия файла и контроль факта отправки. |
| 7 | Внутренний API публикации ставок должен поддерживать работу 24/7 и быстрый отклик за счёт чтения из собственной БД онлайн-контура. |
| 8 | Решение должно использовать технологии банка и подрядчика кол-центра, по которым уже есть внутренняя экспертиза: Java/Spring Boot, MS SQL, Oracle. |
| 9 | Для сопровождения нужны мониторинг обновления ставок, контроль просроченных публикаций и технический аудит по внутреннему API и внешним файлам. |

### <a name="_qmphm5d6rvi3"></a>**Решение**
В качестве базового слоя используется решение из `Task3`: ставки ведутся в АБС, `ABS Adapter` выгружает их в `Deposit DB`, а онлайн-контур не ходит в АБС в пользовательском запросе. Для нового кейса в этот контур добавляются два новых компонента: `Rate Publication Service` для внутреннего кол-центра и `Partner Export Job` для внешнего партнёра.

Состав решения:

- `Rate Publication Service` публикует read-only API с текущими публичными депозитными ставками для системы кол-центра банка;
- `Partner Export Job` собирает внешний файл ставок из того же среза данных и отправляет его партнёрскому кол-центру по защищённому почтовому каналу;
- в системе кол-центра добавляется read-only экран или виджет с текущими ставками;
- в `Deposit DB` хранится единый опубликованный срез ставок и журнал публикаций;
- АБС остаётся единственным местом редактирования правил ставок и специальных условий.

Диаграммы:

- Контекст C4: [call-center-rates-context.puml](call-center-rates-context.puml)
- Компоненты C4: [call-center-rates-components.puml](call-center-rates-components.puml)

Основные компоненты и интеграции:

- `ABS Rate Registry` в АБС хранит все депозитные ставки и правила специальных условий.
- `ABS Adapter` переносит в `Deposit DB` только опубликованный срез ставок, пригодный для консультаций.
- `Rate Publication Service` предоставляет внутренний REST API для системы кол-центра банка.
- Система кол-центра показывает ставки оператору в своей UI-надстройке и может кэшировать их в собственной PostgreSQL для устойчивости.
- `Partner Export Job` формирует CSV/XLSX-файл с актуальными ставками, добавляет версию публикации, дату действия и служебные реквизиты выгрузки.
- Партнёр получает файл через почтовый канал и загружает его в свою внешнюю систему без прямой интеграции с банком.

Логика выбора решения и технологий:

- В `Task3` уже сформирован промышленный реестр ставок в АБС и read model в `Deposit DB`; это нужно переиспользовать, а не строить параллельный контур ставок.
- Для внутреннего кол-центра REST API удобнее и дешевле, чем доработка АБС или ручная пересылка файлов.
- Для внешнего партнёра файловая интеграция обязательна по условию задачи, поэтому выделен отдельный экспортный компонент вместо попытки навязать API.
- Публикация ставок из `Deposit DB`, а не из АБС, защищает перегруженную БД АБС и сохраняет быстрый отклик.
- Партнёру передаются только публичные ставки; это минимизирует риски утечки персонализированных или специальных условий.

Поток публикации ставок:

1. Сотрудники бэк-офиса ведут ставки и правила в АБС.
2. `ABS Adapter` обновляет опубликованный срез ставок в `Deposit DB`.
3. `Rate Publication Service` отдаёт этот срез системе кол-центра банка по внутреннему API.
4. `Partner Export Job` формирует из того же среза внешний файл, логирует публикацию и отправляет его партнёру.
5. Оператор внутреннего и партнёрского кол-центров консультирует клиента по актуальным ставкам.

Участвующие IT-команды:

- Команда АБС: ведение реестра ставок, правила публикации и интеграционная схема выгрузки.
- Команда нового депозитного контура: `Rate Publication Service`, `Partner Export Job`, журнал публикаций, мониторинг.
- Команда подрядчика системы кол-центра: экран ставок и интеграция с внутренним API.
- Команда инфраструктуры и эксплуатации: защищённый почтовый канал, расписание выгрузок, мониторинг и алерты.
- Команда партнёрского кол-центра: импорт файла и операционный регламент обработки новой версии ставок.

### <a name="_bjrr7veeh80c"></a>**Альтернативы**
1. Дать системе кол-центра прямой доступ к АБС или её API.
Плюсы: меньше промежуточных компонентов, единый источник данных.
Минусы: усиливает нагрузку на перегруженную АБС, повышает связанность, усложняет разграничение доступов и не решает задачу внешнего партнёра.

2. Передавать ставки обоим кол-центрам только файлами.
Плюсы: одна интеграционная схема для всех.
Минусы: неудобно для внутреннего кол-центра, больше задержка обновления, выше риск устаревших ставок в работе операторов.

3. Оставить ручной процесс через XLS и email.
Плюсы: минимум изменений в системах.
Минусы: не масштабируется, не подходит для маркетинговой кампании, не даёт контроля актуальности и создаёт операционные ошибки.

**Недостатки, ограничения, риски**

- В системе появляется ещё один канал публикации данных, который нужно мониторить и сопровождать.
- Партнёрский кол-центр получает ставки не в real time, а пакетно через файл; между обновлениями возможна задержка.
- Необходимо строго контролировать состав выгрузки, чтобы во внешний файл не попали специальные или персонализированные условия.
- Решение опирается на работоспособность `Deposit DB` и корректность read model из `Task3`.
- Для защищённой отправки файлов нужен отдельный операционный регламент с паролями, сроками и подтверждением получения.
