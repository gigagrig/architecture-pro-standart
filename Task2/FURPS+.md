# FURPS+ таблица

В таблицу включены архитектурно значимые требования по депозитному MVP и кредитному направлению, а также несколько дополнительных требований, необходимых для проектирования целевого решения.

| Код | Требования | Комментарий |
|-----|------------|-------------|
| F1 | Сайт должен показывать список доступных депозитов с актуальными ставками для новых клиентов | Требование из задания для канала привлечения новых клиентов |
| F2 | Сайт должен принимать заявку на депозит с номером телефона и Ф. И. О. и передавать её в процесс обработки кол-центром | Определяет интеграцию сайта с внутренним контуром банка |
| F3 | Оператор кол-центра должен видеть заявку на депозит, связываться с клиентом и иметь возможность предложить специальные условия | Влияет на интеграцию с системой кол-центра и модель данных заявки |
| F4 | После заявки на депозит с сайта новый клиент должен быть направлен в отделение для идентификации перед открытием продукта | Ключевое бизнес-ограничение процесса для новых клиентов |
| F5 | Интернет-банк должен показывать клиенту список доступных депозитов, актуальные ставки и персонализированные ставки | Требует выделения сервиса ставок и персонализации |
| F6 | Интернет-банк должен позволять выбрать счёт, сумму депозита и отправить заявку на открытие депозита с подтверждением по СМС-коду | Базовый пользовательский сценарий депозитного MVP |
| F7 | В MVP заявка на депозит из онлайн-каналов должна подтверждаться сотрудником бэк-офиса в АБС | Явно ограничивает автоматизацию на горизонте MVP |
| F8 | Клиент должен получать СМС-уведомления после подтверждения ставки по депозиту и после открытия депозита | Требование к событиям и уведомлениям |
| F9 | Ставки по депозитам должны вестись в промышленной системе, а не только в XLS, с возможностью работы сотрудников депозитного и кредитного бэк-офиса | Убирает ручной Excel-процесс; влияет на выбор мастер-системы ставок |
| F10 | Сайт должен показывать кредитные предложения и принимать кредитную заявку с номером телефона и Ф. И. О. | Требование для целевого кредитного канала продаж |
| F11 | Если новый клиент на сайте дополнительно заполнил паспортные данные, система должна сразу выдать решение об одобрении или отказе по кредиту | Требует синхронного вызова скоринга и внешних данных |
| F12 | После подачи кредитной заявки с сайта клиент должен получать СМС о необходимости прийти в отделение для идентификации и выдачи кредита | Требование к уведомлениям и офлайн-продолжению процесса |
| F13 | Сотрудник отделения должен иметь возможность продолжить в АБС уже существующую кредитную заявку, ранее поданную и предодобренную на сайте | Исключает дублирование заявок между каналами |
| F14 | Интернет-банк должен показывать список кредитных продуктов и предодобренные предложения для клиента | Требует хранения и публикации предодобренных офферов |
| F15 | Интернет-банк должен позволять подать заявку на кредит с указанием счёта зачисления и подтверждением операции по СМС-коду | Базовый пользовательский сценарий кредитного канала |
| F16 | После подачи заявки по предодобренному предложению должен выполняться повторный упрощённый скоринг, если ситуация клиента изменилась | Влияет на оркестрацию кредитного процесса |
| F17 | Клиент должен получать СМС-уведомления об изменении статусов кредитной заявки | Требование к событиям жизненного цикла заявки |
| U1 | Интерфейсы сайта и интернет-банка должны быть максимально удобными для клиента | Явное бизнес-требование |
| U2 | Онлайн-сценарии должны соответствовать корпоративной дизайн-системе, цветам и брендированным элементам банка | Влияет на фронтенд-реализацию и переиспользование UI-компонентов |
| U3 | Пользователь должен видеть понятный результат каждого шага: заявка принята, требуется визит в отделение, ставка подтверждена, продукт открыт, кредит одобрен или отклонён | Добавлено: без прозрачных статусов пользователю сложно завершить длинный омниканальный процесс |
| R1 | Все сервисы онлайн-оформления депозитов и кредитов должны работать 24/7 с доступностью не ниже 99,9% | Прямое нефункциональное требование из задания |
| R2 | Онлайн-каналы должны сохранять работоспособность при сбое основного ЦОД с переключением на резервный ЦОД | Требование из описания текущей инфраструктуры интернет-банка |
| R3 | В новом процессе нужно избежать прямой зависимости интернет-банка от API или БД АБС | Иначе перегруженная АБС станет точкой отказа и ограничением доступности |
| R4 | При обмене заявками и статусами между каналами и внутренними системами не допускается потеря или повторная обработка заявок | Добавлено: финансовые заявки должны обрабатываться надёжно и идемпотентно |
| R5 | Новые интеграции не должны снижать надёжность кредитного конвейера и системы скоринга, которые уже рассчитаны на работу 24/7 | Нужно сохранить надёжность наиболее критичных кредитных систем |
| P1 | Отклик по клиентским операциям и загрузке справочных данных должен измеряться миллисекундами | Явное требование бизнеса к скорости интерфейсов |
| P2 | Решение должно выдерживать рост числа онлайн-заявок за счёт горизонтального масштабирования компонентов онлайн-контура и балансировки нагрузки | Влияет на выбор архитектуры взаимодействия каналов и сервисов |
| P3 | Нагрузка на АБС должна быть минимизирована, чтобы массовая онлайн-подача заявок не ставила под угрозу работоспособность банка | Производительность тесно связана с ограничением перегруженной БД АБС |
| P4 | По кредитной заявке с сайта при наличии достаточных данных клиент должен получать предварительное решение сразу в рамках текущей сессии | Следует из требования "сразу выдать решение" |
| P5 | Заявки на кредит по предодобренным предложениям должны обрабатываться сотрудниками в течение того же рабочего дня | Целевое SLA кредитного процесса |
| P6 | Нельзя нагружать систему кредитного скоринга дополнительными фоновыми предрасчётами скорингов | Прямое ограничение на производительность критичной системы |
| S1 | При проектировании нужно максимально использовать технологии, уже применяемые в банке, и стек, по которому есть внутренняя экспертиза | Явное требование к поддерживаемости решения |
| S2 | Новые решения должны быть совместимы с существующими платформами разработки банка | Влияет на выбор БД, интеграционных средств и runtime-платформ |
| S3 | Реализацию бизнес-логики СМС-уведомлений следует вынести за пределы ядра интернет-банка, чтобы избежать доработки ядра подрядчиком | Поддерживаемость и независимость от подрядчика |
| S4 | Архитектура должна поддерживать дальнейшее расширение и сопровождаться документацией для следующих этапов цифровизации | Явное требование из задания |
| S5 | Для новых процессов нужны централизованные журналирование, мониторинг и трассировка статусов заявок по всем каналам | Добавлено: без этого сопровождение распределённого процесса будет затруднено |
| S6 | Новую функциональность интернет-банка целесообразно изолировать так, чтобы её можно было развивать как отдельные сервисы без переписывания всего монолита | Добавлено: следует из возможного перехода к микросервисной архитектуре в рамках депозитного кейса |
| +R1 | Передача чувствительных данных с сайта и из интернет-банка должна быть защищена шифрованием трафика | Прямое требование безопасности |
| +R2 | Сотрудники депозитного и кредитного бэк-офиса не должны получать доступ к данным друг друга сверх разрешённых границ | Ограничение из текущего процесса и требований безопасности |
| +R3 | Текущий интернет-банк реализован как монолит ASP.NET MVC 4.5 на платформе подрядчика; обновление ядра зависит от подрядчика | Важное архитектурное ограничение текущего ландшафта |
| +R4 | АБС реализована на Delphi/Oracle/PL-SQL, а её база данных уже перегружена и масштабируется только вертикально | Критичное техническое ограничение |
| +R5 | Если для интеграций понадобятся очереди сообщений, предпочтение нужно отдавать Kafka, но текущая платформа интернет-банка с ней несовместима | Ограничение на выбор интеграционного стиля и способ внедрения очередей |
| +R6 | Обмен между АБС и кредитным конвейером сейчас происходит через БД раз в сутки, и ускорить этот обмен нельзя | Ключевое ограничение для кредитной архитектуры |
| +R7 | Для нового клиента в сценарии кредита с сайта доступны только данные из анкеты клиента и бюро кредитных историй; внутреннего банковского профиля ещё нет | Ограничение на состав данных для скоринга |
| +R8 | Новый клиент не может быть полностью обслужен удалённо без личной идентификации в отделении банка | Регуляторное и процессное ограничение для внешнего канала |
